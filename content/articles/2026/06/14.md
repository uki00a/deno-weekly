## Deno v2.8.3

[Deno v2.8.3](https://github.com/denoland/deno/releases/tag/v2.8.3)がリリースされています。

### `deno compile`

`deno compile`に`--exclude-unused-npm`オプションが追加されています ([#34741](https://github.com/denoland/deno/pull/34741))。実行可能ファイルの作成時にアプリケーションにおいてimportされていないnpmパッケージを実行可能ファイルから除外してくれるようです。[Deno v2.8.2]({{< ref "articles/2026/06/07.md" >}})で実装された`deno compile --bundle`と同様に、実行可能ファイルのサイズの削減が期待されそうです。

また、`deno compile`で`--watch`がサポートされています ([#34860](https://github.com/denoland/deno/pull/34860))

### `deno x`

[`deno x`]({{< ref "articles/deno/v2.6.md" >}})コマンドに`--ignore-scripts`オプションが追加されています ([#34952](https://github.com/denoland/deno/pull/34952))。このオプションに引数を指定しない場合、すべてのnpmパッケージのライフサイクルスクリプトが無効化されます。引数が指定された場合、指定されたnpmパッケージに対してのみライフサイクルスクリプトの実行が無効化されます。

### `deno lsp`

以下の改善が実施されています:

- `Deno.test()`に対する`Debug` code lens がサポートされています ([#34742](https://github.com/denoland/deno/pull/34742))
- `textDocument/codeAction`: `Deno.test()`に対する`ignore`/`only`オプションのトグルがサポートされています ([#34861](https://github.com/denoland/deno/pull/34861))
- `deno.json`の`importMap`で指定されたImport mapファイルに対する診断がサポートされています ([#34864](https://github.com/denoland/deno/pull/34864))
- LSPが[`deno doc --lint`]({{< ref "articles/deno/v1.38.md" >}})及び`deno publish --dry-run`相当の診断を実施してくれるように改善されています。JSRパッケージ中の各エントリーポイントにおける公開APIのJSDocコメントや型定義について検査してくれます ([#34733](https://github.com/denoland/deno/pull/34733), [#34740](https://github.com/denoland/deno/pull/34740))

### `deno test --doc`

[`deno test --doc`]({{< ref "articles/2024/09/22.md" >}})がコードブロック中のshebangを解釈してくれるように改善されています。shebangで指定されたパーミッションフラグに基づいて`Deno.test()`の`permissions`オプションが自動で設定されます ([#35052](https://github.com/denoland/deno/pull/35052))

### `deno fmt`

[`json.trailingCommas`](https://github.com/dprint/dprint-plugin-json/blob/ac8b204bc6128b06edbd340057db58bace993728/deployment/schema.json#L85-L102)オプションが追加されています ([#33383](https://github.com/denoland/deno/pull/33383))

```json
{
  "fmt": {
    "json.trailingCommas": "jsonc"
  }
}
```

### `deno add`

`deno add`や`deno update`などの各種依存関係管理用のコマンドで[`--env-file`]({{< ref "articles/deno/v1.46.md" >}})オプションがサポートされています ([#34843](https://github.com/denoland/deno/pull/34843))。`.env`で`DENO_AUTH_TOKENS`が定義されているようなケースなどが想定されています。

また、パッケージインストール時のバージョンの範囲指定がサポートされています ([#34799](https://github.com/denoland/deno/pull/34799))

```shell
$ deno add 'jsr:@david/dax@>=0.48.0'
```

### `deno outdated`/`deno update`

[`catalog`/`catalogs`]({{< ref "articles/deno/v2.8.md" >}})で定義されたパッケージを認識してくれるように改善されています ([#34977](https://github.com/denoland/deno/pull/34977))

### `deno bump-version`

[`deno bump-version`]({{< ref "articles/deno/v2.8.md" >}})コマンドに`--config`オプションが追加されています ([#34770](https://github.com/denoland/deno/pull/34770))。`package.json`と`deno.json`が併用されたプロジェクトにおいて`--config package.json`を指定することにより、`deno.json`ではなく`package.json`の`version`を更新したいケースでの使用が想定されています。

### `deno bundle`

`deno bundle`コマンドで`--check`オプションがサポートされています ([#33514](https://github.com/denoland/deno/pull/33514))。型チェックを実施してからバンドルしてくれます。

### `deno clean`

[`--except`]({{< ref "articles/deno/v2.3.md" >}})の指定なしでの`deno clean --dry-run`の実行がサポートされています ([#34846](https://github.com/denoland/deno/pull/34846))

```shell
$ deno clean --dry-run
Removed /path/to/.deno (2737 files, 98.07MB)
Aborting due to --dry-run flag
```

### `deno publish`

`deno publish`コマンドで[`with { type: "text" }`]({{< ref "articles/deno/v2.4.md" >}})を含むパッケージがサポートされています ([#34692](https://github.com/denoland/deno/pull/34692))

### `deno jupyter`

Jupyterカーネルにおいて受信したメッセージのHMAC署名が検証されるように改善されています ([#34825](https://github.com/denoland/deno/pull/34825))

### `links`

`deno.json`の[`links`]({{< ref "articles/2025/06/15.md" >}})においてglobがサポートされています ([#34849](https://github.com/denoland/deno/pull/34849))。パターンに基づいて複数のローカルパッケージを一括で指定できます。

### パーミッション

`--deny-read`または`--deny-write`に指定されたパスよりも祖先のパスに対する読み書きが拒否されてしまう問題が修正されています ([#34504](https://github.com/denoland/deno/pull/34504))

`Deno.stat()`などのAPIで`/proc`などの特殊ファイルを読もうとすると、`--allow-read`しか要求されない問題が修正されています ([#34393](https://github.com/denoland/deno/pull/34393))

また、`Worker`に`deno.permissions`オプションが設定されている場合、その`Worker`内で静的に読み込み先モジュールを特定できない[text/bytesファイルに対する`import()`]({{< ref "articles/deno/v2.4.md" >}})を使用している場合、`deno.permissions`オプションではなく親のパーミッションが`Worker`に適用されてしまう問題が修正されています ([#34707](https://github.com/denoland/deno/pull/34707))

### ワークスペース

[ワークスペース]({{< ref "articles/deno/v1.45.md" >}})中の`package.json`を持つメンバーにて定義されたライフサイクルスクリプトが実行されるように改善されています ([#34615](https://github.com/denoland/deno/pull/34615))

### Deno API

#### `Deno.serve()`

HTTP/1.1向けの実装における[hyper](https://github.com/hyperium/hyper)への依存が削除されています ([#34446](https://github.com/denoland/deno/pull/34446), [#35049](https://github.com/denoland/deno/pull/35049))。これによりパフォーマンス改善などが期待されるようです。

#### FFI

FFI 関連のAPIで`Uint8Array<ArrayBufferLike>`型の値がサポートされています ([#34999](https://github.com/denoland/deno/pull/34999))

#### `Deno.listen()`/`Deno.listenDatagram()`

`Deno.listen()`及び`Deno.listenDatagram()`において`port`または`path` (`transport: "unixpacket"`) の省略がサポートされています ([#31681](https://github.com/denoland/deno/pull/31681))。`port`が省略された場合は`0`がデフォルトで使用されます。`path`が省略された場合は送信専用のソケットが作成されます。

#### Stream

`Deno.stdin.readable`などの特定リソースに紐づく`ReadableStream`をキャンセルした際に、 読み込みが適切にキャンセルされるように改善されています ([#34986](https://github.com/denoland/deno/pull/34986))

また、`Deno.Conn.writable`などの特定リソースに紐づく`WritableStream`へ任意の`ArrayBuffer`または`ArrayBufferView`の書き込みがサポートされています ([#35020](https://github.com/denoland/deno/pull/35020))

### Web API

#### `BroadcastChannel`

`BroadcastChannel`間での`SharedArrayBuffer`のやり取りがサポートされています ([#34961](https://github.com/denoland/deno/pull/34961))

#### Fetch API

`Request`コンストラクターで`priority`オプションがサポートされています ([#34716](https://github.com/denoland/deno/pull/34716))

#### Web Crypto API

[WICG/webcrypto-modern-algos](https://github.com/WICG/webcrypto-modern-algos)に関連して下記の改善が実施されています:

- `SubtleCrypto.supports()`が実装されています ([#34903](https://github.com/denoland/deno/pull/34903))
- `ML-DSA-44`/`ML-DSA-65`/`ML-DSA-87`: `SubtleCrypto#importKey()`及び`SubtleCrypto#exportKey()`においてJWK形式がサポートされています([#34914](https://github.com/denoland/deno/pull/34914))
- `getPublicKey()`メソッドが`CryptoKey`から`SubtleCrypto`へ移動しています ([#34913](https://github.com/denoland/deno/pull/34913))
- `ChaCha20-Poly1305`: `importKey()`/`exportKey()`におけるrawフォーマットが廃止され、代わりに`raw-secret`フォーマットがサポートされています。また、`encrypt()`/`decrypt()`の`nonce`パラメーターが削除され、代わりに`iv`が追加されています ([#34915](https://github.com/denoland/deno/pull/34915))
- `ML-KEM-512`/`ML-KEM-768`/`ML-KEM-1024`: `SubtleCrypto#importKey()`及び`SubtleCrypto#exportKey()`において`raw-seed`及び`jwk`形式がサポートされています ([#34924](https://github.com/denoland/deno/pull/34924))
- `ML-DSA-44`/`ML-DSA-65`/`ML-DSA-87`における`raw-private`フォーマットでの`SubtleCrypto#importKey()`/`SubtleCrypto#exportKey()` 及び `SHAKE128`/`SHAKE256`のサポートが削除されています (`cSHAKE128`/`cSHAKE256`にて代替可能なため) ([#34941](https://github.com/denoland/deno/pull/34941))
- `raw-public`フォーマットが`raw`のエイリアスとして機能するように改善されています ([#34920](https://github.com/denoland/deno/pull/34920))

## Denoの直近の開発について

まだマージはされていませんが、直近でいくつか面白そうなPRが作成されているため、紹介いたします。

### `"jsxImportSource": "deno-jsx:preact"`

まだマージされていませんが、設定なしでJSXを動作させるための機能を導入するPRが作成されています:

- [feat(run/serve): zero-config JSX with an in-binary Preact runtime #35194](https://github.com/denoland/deno/pull/35194)

現時点においては、ユーザーが`jsxImportSource`を設定していない場合に限り、`jsxImportSource`のデフォルト値として[`deno-jsx:preact`](https://github.com/denoland/deno/blob/dec0e5d14d11c38a7ac50ed7bbd01baecfc012ff/libs/resolver/deno_json.rs#L251)が設定されます。`deno-jsx:preact`が設定されている場合、Deno本体に組み込まれたPreactベースの内部モジュールを使用することで機能が実現される想定のようです。

本機能については[2.9.0](https://github.com/denoland/deno/milestone/73)のマイルストーンに含まれています。

### `Deno.McpServer` & `Deno.S3Client`

**Draft状態のため、マージされるかどうかはまだわかりません**が、`Deno.McpServer`及び`Deno.S3Client`を実装するPRが作成されています:

- [feat(unstable): `Deno.McpServer` API for writing MCP servers #35178](https://github.com/denoland/deno/pull/35178)
- [feat(unstable): Deno.S3Client and Deno.s3 #35180](https://github.com/denoland/deno/pull/35180)

`Deno.McpServer`はMCPサーバーを実装するためのAPIで利用するためには`--unstable-mcp`の指定が必要な想定のようです。

もう一方の`Deno.S3Client`はS3互換のオブジェクトストレージ向けのクライアントで、利用するためには`--unstable-s3`の指定が必要なようです。
